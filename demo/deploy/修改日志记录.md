## 2026-05-30 摄像头实时检测无窗口模式修复

### 修改背景

- 板端运行 `camera` 命令时，摄像头可以成功打开，例如 `成功使用V4L2接口打开摄像头`。
- 程序随后在 `cv::namedWindow` 处崩溃，错误信息显示当前打包的 OpenCV 3.4.5 没有 GTK/Qt/Carbon 等 HighGUI 窗口后端。
- 崩溃发生在预览窗口创建阶段，不是摄像头节点、YOLO 模型、RKLLM 大模型或 KG 规则库的问题。

### 模块化改动

- 新增 `src/input/preview_window.h` 和 `src/input/preview_window.cpp`。
- 将 OpenCV HighGUI 相关能力封装为 `input::PreviewWindow` 模块。
- `PreviewWindow` 统一负责：
  - 读取 `DEMO_PREVIEW` 环境变量；
  - 在 `DEMO_PREVIEW=1` 时才尝试创建预览窗口；
  - 捕获 `cv::namedWindow` / `cv::imshow` / `cv::waitKey` 的异常；
  - 在窗口后端不可用时自动切换为无预览模式；
  - 保证摄像头和视频检测主流程不会因为显示功能失败而中断。
- 修改 `src/yolov8_utils.cpp`：
  - `camera` 实时检测流程改为复用 `input::PreviewWindow`；
  - `video` 本地视频检测流程也改为复用同一预览模块；
  - 创建视频文件失败时，日志改为“继续检测但不保存视频”，避免误导为必须依赖预览。
- 修改 `CMakeLists.txt`：
  - 将 `src/input/preview_window.cpp` 加入 `demo` 可执行程序编译列表。

### 模块功能边界

`input::PreviewWindow` 是输入侧辅助模块，不是检测模块，也不是推理模块。

负责：

- 管理 OpenCV 预览窗口生命周期。
- 读取 `DEMO_PREVIEW` 判断是否启用预览。
- 处理窗口创建、显示、按键、关闭。
- 捕获板端无窗口后端导致的 OpenCV 异常。
- 在预览不可用时保证检测流程继续运行。

不负责：

- 不选择摄像头节点，摄像头节点仍由 `DEMO_CAMERA` 和 `open_camera()` 控制。
- 不读取帧，帧读取仍在 `process_camera_stream()` / `process_local_video()` 中。
- 不执行 YOLO 推理，推理仍由 `inference_yolov8_model()` 执行。
- 不执行 Qwen2-VL/RKLLM 推理。
- 不负责 KG 规则分析。
- 不负责 HTTP 上传。

### 注释规范

- 新增模块已补充详细中文注释，说明模块职责、模块边界、环境变量语义和异常降级原因。
- `yolov8_utils.cpp` 的调用点已补充中文注释，说明为什么检测流程只调用预览模块，而不直接调用 `namedWindow/imshow`。
- `src/input/README.md` 已补充中文模块说明，后续双光/融合输入相关功能可以继续按该模块边界扩展。

### 运行方式

无窗口模式，适合板端 SSH/串口/无桌面环境：

```bash
cd /home/elf/project/demo/deploy/install/demo_Linux_aarch64
export LD_LIBRARY_PATH=$PWD/lib:$LD_LIBRARY_PATH
export DEMO_CAMERA=/dev/video21
export DEMO_PREVIEW=0
DEMO_UPLOAD_MODE=offline DEMO_MAX_NEW_TOKENS=32 ./demo
```

进入交互界面后：

```text
camera
```

如果板端确认有可用桌面和 OpenCV HighGUI 后端，可以尝试打开预览：

```bash
export DEMO_PREVIEW=1
```

### 验证重点

- `camera` 命令应能在无预览模式下继续执行 YOLO 实时检测，不再因为 `cvNamedWindow` 崩溃。
- `camera_shot` 仍然用于验证完整慢链路：摄像头拍照、YOLO 检测、KG 提示词、大模型推理。
- 当前修改只修复单路摄像头预览崩溃问题；双光同步采集、SeaFusion 融合、融合图再送 YOLO/VLM 仍是后续独立功能。

### 构建注意

- `build-linux.sh` 会重新生成 `install/demo_Linux_aarch64`。

  - `Qwen2-VL-2B_vision_rk3588.rknn`

## 2026-05-30 实时检测日志降噪与视频保存降级

### 修改背景

- 板端执行 `camera` 后，终端持续打印大量 `rknn_run`。
- `rknn_run` 不是错误，而是 YOLO/RKNN 检测模块每帧推理前的调试输出。
- `camera` 命令确实是在获取摄像头视频流，并对每帧执行 YOLO 检测。
- “无法创建视频文件”只表示检测后的画面无法编码保存到本地文件，不表示摄像头视频流获取失败。

### 模块化改动

- 修改 `src/rknpu2/yolov8.cc`：
  - 将逐帧 `printf("rknn_run")` 改为默认关闭；
  - 新增 `DEMO_VERBOSE_RKNN` 调试开关；
  - 只有设置 `DEMO_VERBOSE_RKNN=1` 时，才逐帧打印 `rknn_run`。
- 新增 `src/input/video_recorder.h` 和 `src/input/video_recorder.cpp`。
- 将视频保存能力封装为 `input::openVideoRecorder()`。
- 修改 `src/yolov8_utils.cpp`：
  - `camera` 视频流检测复用 `input::openVideoRecorder()`；
  - `video` 本地视频检测也复用同一保存模块；
  - 视频保存失败时只影响证据文件，不影响实时检测主循环。
- 修改 `src/main.cpp`：
  - 将 `camera` 菜单说明改为“获取视频流并实时检测”；
  - 明确 10 秒自动结束，只有 `DEMO_PREVIEW=1` 且预览窗口可用时才支持 ESC 退出。
- 修改 `CMakeLists.txt`：
  - 将 `src/input/video_recorder.cpp` 加入 `demo` 可执行程序编译列表。

### 模块功能边界

`camera` 命令包含两层含义：

- 实时视频流处理：打开摄像头、读取连续帧、逐帧执行 YOLO/RKNN 检测。
- 可选结果保存：把已经画好检测框和 FPS 的帧写入本地视频文件。

`input::VideoRecorder` 只负责第二层，也就是“创建可写的视频文件”。

负责：

- 优先尝试首选路径和 MP4V 编码。
- 如果 MP4 不可写，自动降级为同名 `.avi` 和 MJPG 编码。
- 返回实际保存路径和编码名称。

不负责：

- 不打开摄像头。
- 不读取视频流。
- 不执行 YOLO/RKNN 检测。
- 不显示预览窗口。
- 不执行 RKLLM 大模型推理。
- 不上传结果。

### 运行方式

默认安静模式：

```bash
export DEMO_VERBOSE_RKNN=0
```

需要确认每帧是否进入 RKNN 推理时，再打开调试：

```bash
export DEMO_VERBOSE_RKNN=1
```

### 验证重点

- `camera` 仍应持续获取摄像头视频流并逐帧检测。
- 默认情况下终端不应再被 `rknn_run` 刷屏。
- 如果 MP4 编码不可用，程序应自动尝试生成 `.avi` 文件。
- 如果所有视频编码都不可用，程序仍应继续实时检测，只是不保存视频证据文件。

## 2026-05-30 部署包 libcurl 依赖策略修复

### 修改背景

- 新版 `install/demo_Linux_aarch64` 传到板端后，执行 `./demo` 报错：
  - `libldap_r-2.4.so.2: cannot open shared object file`
- 该问题不是 SeaFusion、YOLO、Qwen 或摄像头代码错误，而是部署包里的旧 `libcurl.so.4` 被优先加载。
- `demo` 的运行时搜索路径包含 `$ORIGIN/lib`，如果 `install/lib` 内存在交叉环境打包来的 `libcurl.so.4`，动态链接器会优先使用它。
- 交叉环境的 `libcurl.so.4` 可能依赖旧系统库，例如 `libldap_r-2.4.so.2`，而板端 Ubuntu 22.04 默认不一定提供该旧库。

### 模块化改动

- 修改 `CMakeLists.txt` 的部署依赖策略：
  - 编译和链接阶段仍然使用 `libcurl`，保留 `send_info` 在线上传能力；
  - 默认不再把交叉环境里的 `libcurl.so*` 安装到 `install/demo_Linux_aarch64/lib`；
  - 让板端使用系统自带的 `libcurl4`，避免旧 `libcurl` 间接依赖缺失。
- 新增 CMake 选项：
  - `DEMO_BUNDLE_LIBCURL=OFF`：默认策略，不打包 `libcurl.so*`；
  - `DEMO_BUNDLE_LIBCURL=ON`：仅在需要离线全量打包时使用，同时必须补齐 `libcurl` 的所有间接依赖。

### 后续验证重点

- 重新执行 `./build-linux.sh` 后，`install/demo_Linux_aarch64/lib` 中不应再出现：
  - `libcurl.so`
  - `libcurl.so.4`
  - `libcurl.so.4.6.0`
- 板端如未安装系统 curl 运行库，需安装一次：
  - `sudo apt install -y libcurl4`
- 板端执行前建议先检查：
  - `ldd ./demo | grep "not found"`

- 这里的 `No space left on device` 通常不是磁盘空间不足，而是双路摄像头在当前分辨率、帧率或像素格式下占用的总线带宽过高。

### 模块化改动

- 修改 `src/camera.h`：
  - 将默认采集帧率从 `30fps` 调整为 `15fps`；
  - 在中文注释中说明双路 RGB/IR 同时取流的带宽风险。
- 修改 `src/camera.cpp`：
  - 新增摄像头采集参数环境变量解析；
  - 参数解析集中在 camera 模块，不散落到 YOLO、SeaFusion 或 main.cpp。
  - 双路取帧失败后释放 RGB/IR `VideoCapture` 句柄并重置打开状态，避免一次 `VIDIOC_STREAMON` 失败后同一进程内持续 `VIDIOC_QBUF: Invalid argument`。

### 新增环境变量

- `DEMO_CAMERA_WIDTH`：覆盖采集宽度，默认 `640`；
- `DEMO_CAMERA_HEIGHT`：覆盖采集高度，默认 `480`；
- `DEMO_CAMERA_FPS`：覆盖采集帧率，默认 `15`；
- `DEMO_RGB_CAMERA`：覆盖 RGB 摄像头节点；
- `DEMO_IR_CAMERA`：覆盖 IR 摄像头节点；
- `DEMO_CAMERA`：覆盖单路实时检测摄像头节点。

## 2026-05-30 新增 camera_loop 准实时持续拍照模式

### 修改背景

比赛展示需要区分三种模式：

```text
拍一张：完整单次闭环，适合展示大模型分析。
持续拍照：低频连续刷新，适合PC端Qt准实时展示。
实时视频：板端连续视频流检测，后续可升级为网络视频流推送。
```

原有 `camera_shot` 每次都会触发 Qwen2-VL/RKLLM，大模型耗时较长，
不适合逐帧运行。为了保证 Qt 界面持续刷新，新增轻量的 `camera_loop`。

### 修改内容

- 修改 `src/main.cpp`：
  - 新增 `CameraLoopOptions`。
  - 新增 `parse_camera_loop_options()`。
  - 新增 `build_camera_loop_upload_text()`。
  - 新增 `run_camera_loop_upload()`。
  - 菜单新增：

```text
[camera_loop [次数] [间隔ms]] 准实时循环采集/融合/检测/四图上传 (默认10帧/2000ms)
```

### 三种模式边界

```text
camera_shot
    拍一张。
    采集RGB/IR -> SeaFusion融合 -> YOLO检测 -> KG -> Qwen2-VL分析 -> 四图上传。
    适合单次完整AI闭环展示。

camera_loop [次数] [间隔ms]
    持续拍照。
    采集RGB/IR -> SeaFusion融合 -> YOLO检测 -> KG轻量文本 -> 四图上传。
    不逐帧运行Qwen2-VL，保证PC端Qt能持续刷新。

camera
    实时视频。
    当前是板端RGB视频流YOLO检测，10秒自动结束。
    目前不等价于PC端Qt视频流推送。
```
